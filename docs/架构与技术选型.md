# CtxForge 架构与技术选型

## 1. 项目定位

CtxForge 不是完整 coding agent，而是一个 DeepSeek-native 的 Memory + Context Engineering Runtime。

它的核心目标是提供一套可以被其他 Agent 复用的底层能力：

- 分层 Memory：工作记忆、会话摘要、长期向量记忆。
- Context Builder：稳定构造 prompt，管理 token 预算，区分稳定 prefix 与动态内容。
- Prefix-cache Observatory：追踪上下文变化，估算 cache hit/miss，输出可解释报告。
- Skill System：用文件协议定义、安装、发现、激活技能。
- Light Agent Demo：用 CLI/TUI 演示 runtime 的完整调用链。

ClaudeCode 的调研结论说明，一个成熟 CLI Agent 的复杂度主要来自入口、TUI、工具、权限、MCP、插件、状态管理和模型循环的交叉。CtxForge 第一阶段不复刻这些完整能力，而是借鉴它的分层方式，把可复用 runtime 做清楚。

## 2. 借鉴 ClaudeCode 的设计原则

| ClaudeCode 方案 | CtxForge 借鉴方式 | 收敛策略 |
| --- | --- | --- |
| 多入口快速路径 | CLI 命令分层，快速命令不加载 TUI/向量库 | MVP 只保留 `run`、`inspect`、`memory`、`skill`、`config` |
| Command/Skill/MCP prompt 统一 | 把用户命令、skill、模板统一成 `CommandSpec` | 第一版不接 MCP，只保留本地文件协议 |
| Tool 抽象携带安全元数据 | 将 runtime 能力声明为 `RuntimeTool`，包含只读、写入、预算影响 | 不开放任意 shell 工具 |
| Query loop 编排模型和工具 | 设计轻量 `AgentLoop` 调用 context、memory、DeepSeek | 不做复杂工具递归和大规模代码编辑 |
| settings 级联 | 支持 global/project/local 配置 | 不做 managed policy 和企业策略 |
| 文档式插件/技能 | Skill 使用 `SKILL.md` + manifest | 不做 marketplace，先做本地安装 |
| TUI AppState | TUI 只展示上下文、memory、cache、响应状态 | 不做复杂权限弹窗和远程 session |

## 3. 总体架构

```text
User
  -> CLI / TUI
      -> Command Router
          -> Agent Runtime
              -> Skill Registry
              -> Memory Manager
              -> Context Builder
              -> Prefix Cache Analyzer
              -> DeepSeek Client
              -> Observability Reporter
          -> Session Store
```

核心分层：

1. Interface Layer
   - 提供 CLI、TUI 和 headless API。
   - 只负责输入输出、命令路由和展示。

2. Runtime Layer
   - 提供 CtxForge 的核心编排。
   - 调度 memory、skill、context、model、report。

3. Context Engineering Layer
   - 管理上下文分区、token 预算、prefix 稳定性和 cache 报告。

4. Memory Layer
   - 负责短期状态、会话摘要、长期向量记忆。

5. Extension Layer
   - 负责 skill 文件协议、skill 发现、安装和激活。

6. Infrastructure Layer
   - 负责配置、日志、存储、DeepSeek API、tokenizer、向量库。

## 4. 模块设计

### 4.1 CLI / TUI

职责：

- 接收用户任务。
- 调用 runtime。
- 展示模型响应、上下文报告、memory 命中、cache 变化。
- 提供基础管理命令。

建议命令：

```text
ctxforge run "task"
ctxforge tui
ctxforge inspect context
ctxforge inspect cache
ctxforge memory list
ctxforge memory search "query"
ctxforge skill list
ctxforge skill install ./path/to/skill
ctxforge config show
```

边界：

- CLI 不直接拼 prompt。
- TUI 不直接读写 memory。
- 所有核心逻辑都走 runtime API。

### 4.2 Command Router

职责：

- 把 CLI/TUI 输入转换为内部命令。
- 加载可用 skill command。
- 做基础参数校验。

核心类型：

```python
class CommandSpec:
    name: str
    description: str
    source: Literal["builtin", "skill"]
    allowed_runtime_tools: list[str]
    context_template: str | None
```

借鉴点：

- ClaudeCode 把 slash command、skill、plugin command、MCP prompt 统一成 Command。
- CtxForge 第一版只统一 builtin command 和 local skill command。

### 4.3 Agent Runtime

职责：

- 组织一次用户请求的完整链路。
- 明确每一步产物，方便调试和可观测。

MVP 链路：

```text
User Task
  -> normalize input
  -> select skills
  -> load working memory
  -> retrieve long-term memory
  -> build context
  -> analyze prefix cache
  -> call DeepSeek
  -> generate session summary
  -> persist memory
  -> emit response + report
```

核心类型：

```python
class RuntimeRequest:
    task: str
    cwd: Path
    session_id: str | None
    skill_names: list[str]
    max_tokens: int | None

class RuntimeResult:
    answer: str
    context_report: ContextReport
    cache_report: CacheReport
    memory_report: MemoryReport
    session_id: str
```

### 4.4 Memory Manager

分三层：

| 层级 | 存储内容 | 生命周期 | 存储方案 |
| --- | --- | --- | --- |
| Working Memory | 当前任务、临时事实、todo、最近工具结果摘要 | 单 session | SQLite / JSON |
| Session Summary | 每轮会话压缩摘要、关键决策、用户偏好变化 | 跨 session | SQLite |
| Long-term Vector Memory | 项目事实、用户偏好、历史决策、可检索经验 | 长期 | 纯 SQLite + 余弦相似度，后续验证 sqlite-vec |

Memory 写入规则：

- 默认不把完整对话全部写入长期记忆。
- 长期记忆只保存可复用事实、决策、偏好和项目知识。
- 每条长期记忆必须带来源、时间、作用域和置信度。

核心类型：

```python
class MemoryRecord:
    id: str
    scope: Literal["global", "project", "session"]
    kind: Literal["preference", "fact", "decision", "summary"]
    content: str
    source: str
    created_at: datetime
    confidence: float
    embedding: list[float] | None
```

### 4.5 Context Builder

职责：

- 把 system prompt、runtime rules、skill、memory、用户任务拼成最终上下文。
- 控制 token 预算。
- 保持 prefix 字节稳定。
- 输出上下文组成报告。

上下文分区：

```text
Stable Prefix
  - runtime system prompt
  - context protocol
  - stable tool/skill manifests

Semi-stable Project Context
  - project profile
  - selected skill docs
  - user preferences

Dynamic Suffix
  - current task
  - retrieved memories
  - session working memory
  - recent summary
```

关键规则：

- Stable Prefix 中的内容必须按确定性顺序渲染。
- 字典、manifest、skill 列表必须排序。
- 时间戳、随机 ID、临时路径不得进入 Stable Prefix。
- 动态内容只能进入 suffix。

### 4.6 Prefix Cache Analyzer

职责：

- 对比当前 context 与上一轮 context。
- 找出 prefix 首个变化位置。
- 估算稳定 token、变化 token、可能 cache hit ratio。
- 标记破坏 prefix 稳定性的模块。

MVP 不直接读取 DeepSeek 真实账单，而是输出估算报告：

```text
stable_prefix_tokens: 4200
changed_after_byte: 16832
estimated_cache_hit_ratio: 0.73
invalidated_sections:
  - selected_skill_manifest
  - project_context
```

后续如果 DeepSeek API 返回 cache hit 元数据，再接入真实指标。

### 4.7 Skill System

目录协议：

```text
skills/
  code-review/
    skill.toml
    SKILL.md
    examples/
    scripts/
```

`skill.toml` 示例：

```toml
name = "code-review"
version = "0.1.0"
description = "Review code changes with project context."
allowed_runtime_tools = ["memory.search", "context.read"]
activation = ["review", "diff", "bug"]
```

职责：

- 发现本地 skill。
- 校验 manifest。
- 加载 `SKILL.md`。
- 根据用户任务做简单激活。
- 将 skill 内容注入 context。

边界：

- 第一版不执行 skill 中的任意脚本。
- `scripts/` 只作为未来扩展保留。
- 不做远程 marketplace。

### 4.8 DeepSeek Client

职责：

- 封装 DeepSeek Chat API。
- 支持 streaming 和 non-streaming。
- 记录 request id、model、token usage、错误。
- 为 prefix cache 预留 headers/metadata 解析点。

设计要求：

- 业务层不直接调用 HTTP SDK。
- API 请求和响应要可序列化，便于调试。
- 模型能力以 `ModelProfile` 描述，避免散落在代码中。

### 4.9 Observability Reporter

输出三类报告：

1. Context Report
   - 各 section token 占用。
   - 被截断内容。
   - 注入的 skill 和 memory。

2. Cache Report
   - 稳定 prefix 字节数。
   - 首个变化 section。
   - 估算 cache hit ratio。

3. Memory Report
   - 检索 query。
   - 命中记录。
   - 写入记录。

这些报告是 CtxForge 和普通 Agent 的差异点，应当成为 TUI 的第一屏核心信息。

## 6. 技术选型

### 6.1 主语言：Python

选择：Python 3.12+

理由：

- 更适合快速实现 memory、向量检索、LLM API、CLI/TUI。
- 生态里已有成熟 tokenizer、SQLite、向量库、配置、测试工具。
- 对学习 Harness 工程来说，核心价值在 runtime 边界和上下文策略，不在前端复杂度。

不选 TypeScript 的原因：

- ClaudeCode 使用 TypeScript + React/Ink 是为了复杂 TUI 和大型 CLI 产品。
- CtxForge MVP 更偏研究框架和 runtime，Python 迭代成本更低。

### 6.2 CLI：Typer

选择：Typer + Rich

理由：

- Typer 基于 type hints，命令定义清晰。
- Rich 可以直接渲染表格、树、diff、进度和报告。
- 未来引入 Textual 时，Rich 的展示模型和终端风格可以继续复用。

### 6.3 TUI：Phase 6 再引入 Textual

选择：Phase 0-5 不引入 TUI 框架，只用 Rich；Phase 6 再引入 Textual。

理由：

- Python 原生，适合展示 memory/context/cache 面板。
- 比 curses 更现代，比 React/Ink 更贴合 Python runtime。
- 支持异步任务、日志面板、表格和可交互视图。

MVP 可以先只做 CLI，TUI 放在第二阶段；但架构上保留 TUI 层。

### 6.4 配置：TOML + Pydantic

选择：

- `ctxforge.toml`：项目配置。
- `~/.ctxforge/config.toml`：用户配置。
- Pydantic：schema 校验。

配置优先级：

```text
default < user < project < env < cli flag
```

理由：

- TOML 可读性强，适合工具配置。
- Pydantic 能给出明确校验错误。
- 借鉴 ClaudeCode settings cascade，但第一版不做 managed policy。

### 6.5 存储：SQLite

选择：Python 内置 `sqlite3`

理由：

- 单机 CLI 工具最稳妥。
- 适合 session、summary、memory metadata、cache snapshot。
- 易备份、易调试、无需服务进程。
- 零额外依赖，第一版直接写 SQL 足够清晰。

SQLAlchemy Core 暂不引入。等 schema 变复杂、迁移需求明确后再评估。

### 6.6 向量记忆：先纯 SQLite，后验证 sqlite-vec

优先选择：先验证 `sqlite-vec` 能否稳定安装；如果不能，先用纯 SQLite + 手写余弦相似度。

理由：

- 与 SQLite 主存储放在一起是理想状态，但 Windows 和本地安装体验要先验证。
- MVP 的长期记忆规模很小，纯 SQLite 保存 embedding，再用 Python 计算余弦相似度已经够用。
- 这样可以先把 memory 链路跑通，不被向量扩展安装问题阻塞。

备选：LanceDB

适用条件：

- 需要更强向量检索性能。
- memory 数据规模明显增长。
- 需要更成熟的 metadata filter。

MVP 建议先抽象 `VectorStore` 接口，默认实现用普通 SQLite + 余弦相似度；`sqlite-vec` 作为验证通过后的优化实现。

### 6.7 Embedding：DeepSeek 优先，兼容本地 embedding

选择：

- DeepSeek embedding，如果当前 API 支持且稳定。
- 备选：BAAI/bge-small 或 sentence-transformers 本地模型。

注意：

- Embedding provider 必须抽象成接口。
- 不把 embedding 逻辑写死在 Memory Manager。
- 如果 DeepSeek embedding 不可用，不能阻塞主链路。

### 6.8 Tokenizer：tiktoken + provider profile

选择：tiktoken 作为估算工具。

理由：

- DeepSeek tokenization 与 OpenAI tokenizer 不一定完全一致，但 MVP 需要的是预算估算和 section 对比。
- 精确 token usage 以 DeepSeek API 返回值为准。

设计：

- `TokenCounter` 只提供估算。
- `UsageRecord` 保存 API 返回的真实 usage。

### 6.9 DeepSeek API：HTTPX

选择：HTTPX

理由：

- 同时支持 sync/async。
- 超时、重试、streaming 支持清晰。
- 不强绑定第三方 SDK，方便记录底层 request/response。

### 6.10 日志与事件：标准库 logging + Rich 展示

选择：

- 标准库 `logging`：基础日志。
- Rich：CLI 友好展示。

理由：

- Context Engineering 需要可解释事件流。
- 后续可把事件写入 JSONL，用于调试和 replay。
- `structlog` 是锦上添花，不应该成为第一天的依赖。

### 6.11 测试：pytest

选择：pytest

重点测试：

- context section 顺序稳定。
- prefix cache analyzer 对 byte diff 的判断。
- memory 写入和检索。
- skill manifest 校验。
- DeepSeek client mock。

`pytest-asyncio`、`respx` 或 `pytest-httpx` 等到真正实现 async streaming 或 HTTP mock 时再加。

### 6.12 包管理：uv

选择：uv

理由：

- 安装和锁定速度快。
- 适合现代 Python CLI 项目。
- 生成 `pyproject.toml` + lockfile，方便复现。

## 8. MVP 实现顺序

### Phase 0：项目骨架

- `pyproject.toml`
- CLI 入口
- 配置加载
- 基础日志
- 测试框架

### Phase 1：Context Builder

- section 模型
- 稳定排序
- token 预算
- context report
- prefix snapshot

### Phase 2：Memory

- SQLite schema
- session summary
- working memory
- long-term memory metadata
- 简单检索

### Phase 3：Skill

- skill manifest
- `SKILL.md` 加载
- skill discovery
- 简单 activation

### Phase 4：DeepSeek Runtime

- DeepSeek client
- runtime request/result
- end-to-end `ctxforge run`
- session summary 写入

### Phase 5：Cache Report

- prefix byte diff
- section hash
- hit ratio 估算
- `ctxforge inspect cache`

### Phase 6：TUI Demo

- 上下文组成面板
- memory 命中面板
- cache report 面板
- streaming response 面板

Phase 0-5 只使用 Rich 做 CLI 展示，不引入 Textual。Textual 是 Phase 6 的 TUI 选型。

## 9. 关键取舍

### 9.1 不先做完整 coding agent

原因：

- 会被工具执行、权限、代码编辑、shell 沙箱拖入大复杂度。
- 和项目定位冲突。
- 面试展示重点应是 Context Engineering Runtime，而不是复刻 ClaudeCode。

### 9.2 不先接 MCP

原因：

- MCP 是重要扩展点，但第一版会显著增加连接、鉴权、工具 schema 和状态复杂度。
- CtxForge 的 Skill System 可以先用本地文件协议验证核心模型。

后续可以把 MCP prompt/skill 映射到 `CommandSpec` 和 `SkillManifest`。

### 9.3 不追求 token 绝对精确

原因：

- DeepSeek tokenizer 和 API 计费细节可能变化。
- MVP 更需要稳定的预算策略和 cache 变化解释。

策略：

- 本地 token 只做估算。
- API 返回 usage 作为真实记录。

### 9.4 Prefix cache 以字节稳定为第一原则

原因：

- Prefix cache 的关键不是模型语义相似，而是请求前缀内容稳定。
- 因此 Context Builder 必须具备确定性渲染能力。

## 10. 最终技术栈表

| 分类 | 选择 | 说明 |
| --- | --- | --- |
| 语言 | Python 3.12+ | 快速实现 runtime、memory、LLM、TUI |
| 包管理 | uv | 快速、可复现 |
| CLI | Typer + Rich | 类型友好，展示能力强 |
| TUI | Textual | Phase 6 再引入，适合面板化 observability |
| 配置 | TOML + Pydantic | 可读、可校验、可级联 |
| 本地存储 | SQLite | 单机 CLI 最稳妥 |
| DB 访问 | Python 内置 sqlite3 | 第一版直接写 SQL，零额外依赖 |
| 向量库 | 纯 SQLite + 余弦相似度，sqlite-vec 验证后再用 | 先保证可跑，后续优化 |
| LLM Client | HTTPX | 可控、支持 async/streaming |
| Embedding | DeepSeek 优先，本地 bge 备选 | DeepSeek-native，但不强绑定 |
| Token 估算 | tiktoken | 预算估算，真实 usage 用 API 返回 |
| 日志 | logging + Rich | 先用标准库，终端展示保持友好 |
| 测试 | pytest | async/HTTP mock 依赖需要时再加 |

## 11. 一句话架构结论

CtxForge 应该做成一个 Python-first、DeepSeek-native、SQLite-backed 的 Context Engineering Runtime：用确定性 Context Builder 保证 prefix 稳定，用分层 Memory 提供跨会话能力，用文档式 Skill 提供扩展入口，再用 CLI/TUI 把 cache、memory 和 context 的运行状态可视化。
